## 1. 자동화할 수 있는 것들을 자동화하자

### 핵심 아이디어

> 바쁠수록 당장 일을 쳐내는 것보다
>
> 당장은 시간이 조금 더 걸리더라도 **자동화할 수 있는 것들을 자동화하자**

---

### 배포 CI/CD, QA 요청 자동화

**문제 상황**

- 상장 준비로 Github Actions 비용을 아낀다며 CI/CD 마저 끊어져 있던 상태
- QA 담당자와 개발자의 업무 효율 하락

**해결 방법**

1. 경영진을 설득하여 CI/CD 재연결 승인받음
2. `main` 브랜치에 머지 → 개발 환경 자동 배포
3. 배포 완료 시 QA 담당자에게 자동 노티

**기술 구현**

- Github API로 PR 정보를 가져와 브랜치명에서 이슈 Key 추출
- 해당 Jira 티켓에 QA 담당자 자동 태그

```yaml
# GitHub Actions workflow 일부
PR_DATA=$(gh api -X GET "repos/${{ github.repository }}/commits/$COMMIT_SHA/pulls" --jq '.[0]' 2>/dev/null || echo "")

BRANCH_NAME=$(echo "$PR_DATA" | jq -r '.head.ref')
echo "Found PR branch: $BRANCH_NAME"

# 브랜치명에서 이슈 키 추출 (예: feat/fc-1234 → FC-1234)
ISSUE_KEY=$(echo "$BRANCH_NAME" | grep -oiE 'fc-[0-9]+' | tr '[:lower:]' '[:upper:]' || true)

```

**효과**

- 배포 시간 단축
- QA 담당자와 개발자의 업무 효율 향상

---

### 수동으로 해주던 일들을 기능으로

- **관성에 따라 요청이 들어오면 생각 없이 해주던 일들**

**개선 사례 1: 면세/과세 설정 UX 개선**

- **문제**: 백오피스 설정이 복잡해서 오설정 발생 → 개발자가 직접 쿼리 날려서 수정
- **해결**: 백오피스 UX를 직관적으로 개선하여 오설정 최소화

**개선 사례 2: 메인페이지 Redirect 설정 자동화**

- **문제**: 하루의 첫 접속 시 특정 페이지로 redirect하는 기능을 개발자가 수동 설정
- **해결**: 백오피스에서 직접 redirect URL을 등록할 수 있도록 기능 구현
- **기술**: Next.js의 middleware 활용

**개선 사례 3: 컴포저(상세 페이지 편집 기능) 확대 적용**

- **문제**: `"경은님 AI월드 배너 내용 수정 요청드립니다. 가격이 3천원 올라갔어요.."` 같은 요청들
- **해결**: 신규 프로젝트에도 컴포저를 최대한 활용하여 비개발자도 직접 수정 가능하게

**효과**

- 반복적인 단순 작업 요청 대폭 감소
- 사업부의 작업 자율성 향상

## 2. 코드의 품질을 유지하자

### 문제 상황

> PR 리뷰를 받을 수 없는 상황에 일은 일대로 많아지니
>
> 자꾸만 떨어지는 코드의 퀄리티

---

### Gemini 코드리뷰 도입

**배경**

- 혼자서는 코드 리뷰를 받을 수 없는 상황
- 빠르게 개발하다 보니 놓치는 부분들이 생김

**해결**

- AI 코드 리뷰어인 Gemini를 도입하여 자동 리뷰 시스템 도입 (타 팀에서 사용하던 것 연결)
- PR 생성 시 자동으로 코드 리뷰 코멘트 생성

**효과**

- 기본적인 코드 품질 체크 자동화
- 놓칠 수 있는 버그나 개선점 발견

---

### E2E 테스트 도입 (Playwright)

**도입 배경: 전사 장애 사건**

- 한 번 전사 장애가 발생한 경험이 있음
- 혼자서 이런 상황을 더 잘 대비하기 위해 E2E 테스트 필요성을 절감

**구현**

- Playwright를 활용한 E2E 테스트 구축
- 주요 사용자 플로우에 대한 자동화 테스트 작성
- CI/CD 파이프라인에 통합하여 main 머지 전 자동 실행

**효과**

- 배포 전 주요 기능 동작 여부 자동 검증
- 버그 조기 발견
- 혼자서도 안심하고 배포할 수 있는 안전망 확보
- 이를 토대로 여러가지 리팩토링도 더 공격적으로 해볼 수 있었음

---

### 리팩토링과 기술 부채 해결

### Next.js 13 → 16 메이저 업그레이드

**도전 배경**

- E2E 테스트가 있으니 자신감을 가지고 시도

**작업 내용**

- Next.js 13에서 16으로 메이저 버전 업그레이드
- 의존성 패키지들도 함께 최신 버전으로 업데이트
- 주요 Breaking Changes 대응 및 코드 마이그레이션
- codemod 최대한 활용

---

### Web Vitals 최적화 작업

**문제 인식**

- MAU 65만 서비스인데 성능 최적화에 소홀했던 상황
- 특히 메인 페이지의 CLS(Cumulative Layout Shift) 문제

**작업 내용**

1. **메인 페이지 CLS 최소화**
   - 이미지 로딩으로 인한 레이아웃 밀림 현상 개선
   - Skeleton UI 적용으로 로딩 상태 명확히 표시
2. **Core Web Vitals 지표 개선**
   - FCP (First Contentful Paint) 최적화
   - CLS (Cumulative Layout Shift) 개선

**혼자서 이 작업을 한 이유**

- 혼자 남으니 오히려 **서비스 전체를 책임진다**는 마인드가 심화
- 동료가 없으니 직업적인 성취가 부족하다 느꼈고, 이런 부분을 충족
- 회사가 코드의 질을 전혀 신경쓰지 않는 환경에서도 퀄리티를 챙길 줄 아는 능력을 배양

---

### API 에러에 대한 처리 강화

**문제 인식**

- 전사 장애 원인이 백엔드 관리 부재로 인한 것이라고 느꼈음
- redis 메모리가 치솟았고 배치 작업이 많은데, 이에 대한 관리 부재로 생각됨
  - 2시 정각에 메모리가 치솟은 것으로 보아 배치 작업으로 의심 중
- API 에러가 throw 되어버리는 상황

**작업 내용**

- 개발 단계에서 API 에러가 딱히 없었던 부분에도 최대한 에러처리
- 메인페이지의 데이터가 없어도 404가 아닌 빈 배열을 보내주어야 했다고 생각하는데, 에러가 오고 있어 프론트에서 에러 처리를 철저히 하였음
- 데이터를 불러오는 여러 위치에서 **Fallback UI** 제공
- **부분 렌더링**: 일부 데이터만 로드되어도 페이지는 보여줌
- Sentry 연결을 위해 설득 준비중.. (데이터독도 돈아낀다고 제거해버려서)

## 3. 혼자 살아남기 위한 마인드셋

### 우선순위를 명확히 받자

**문제**

- 한번에 4~5명의 이해관계자와 동시에 소통해야 하는 상황
- 모든 요청이 급해 보임

**해결**

- 각 요청의 우선순위를 명확히 확인
- 비즈니스 임팩트와 긴급도를 기준으로 우선순위 정렬
- 다행히도 우선순위 가이드라인이 있었음
