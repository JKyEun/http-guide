## URN (Uniform Resource Name)

**실제 예시:**
- `urn:isbn:0451450523` (책의 ISBN)
- `urn:uuid:6e8bc430-9c3a-11d9-9669-0800200c9a66` (UUID)
- `urn:ietf:rfc:3986` (RFC 문서)

웹에서는 거의 안 쓰임. URL(위치 기반)이 훨씬 실용적. URN은 "이름"만 정의하는데, 실제 위치는 별도로 해결해야 해서 복잡함.

## TCP 커넥션

**1. 실제 동작**

각 계층의 역할:
- **응용 계층**: HTTP 메시지 생성
- **전송 계층(TCP)**: 데이터를 세그먼트로 분할, 순서 번호 부여, 재전송 관리
- **네트워크 계층**: IP 패킷으로 감싸서 경로 찾기
- **링크 계층**: 물리적 전송

**2. 커넥션 과정 (3-way handshake)**
```
클라이언트 → 서버: SYN
서버 → 클라이언트: SYN-ACK
클라이언트 → 서버: ACK
```

## URL 프래그먼트

자주 사용:
- `https://example.com/docs#introduction` - 페이지 내 특정 섹션
- `file.js#L42-L55` - GitHub 코드의 특정 라인
- `#/dashboard` - SPA 클라이언트 라우팅

서버로 전송되지 않고 브라우저에서만 처리.

## URL 구조 설계

### **페이지 접근 관점 (사용자용)**
- 직관적이고 읽기 쉬운 구조 우선
- `/about`, `/products`, `/blog/my-first-post`
- SEO 고려 (의미 있는 단어 사용)
- 계층 구조로 사이트 구조 반영
- 예: `/category/subcategory/item`

### **API 호출 관점 (개발자용)**
- 리소스 중심 설계
- **기본 원칙**: 
  - 명사 사용: `/users`, `/posts`
  - 복수형: `/users/123`
  - 계층 구조: `/users/123/posts`
  - HTTP 메서드로 행위 표현: `GET /users`, `POST /users`, `DELETE /users/123`
- **현실**:
  - 검색/필터: `/articles?author=kim&sort=date`
  - 행위 기반도 허용: `/articles/42/publish`
  - 버전 관리: `/api/v1/users`


## 신뢰성 있는 전송 프로토콜

**1. 신뢰성이란?**
데이터가 손실 없이, 순서대로, 중복 없이 전달되는 것을 보장. TCP는 패킷 유실 시 재전송하고, 순서 섞이면 재조립하고, ACK로 전달 확인.

**2. 비신뢰 프로토콜**
UDP가 대표적. 데이터 전송만 하고 도착 확인 안 함. 실시간 스트리밍, 온라인 게임, DNS 조회에서 사용 (약간의 손실 허용, 속도 중요).
