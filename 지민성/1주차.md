# 1부 1장. HTTP 개관

## HTTP와 URI
- HTTP : HyperText Transfer Protocol
  - 즉 HyperText를 Transfer하기 위한 Protocol
- 이 HTTP는 `인터넷의 멀티미디어 배달부`이다.
- 웹 서버는 이 배달부로, 즉 HTTP 프로토콜로 의사소통하는 `HTTP 서버`이라고 볼 수 있다.
- 일반적으로 `HTTP 서버(웹 서버)`와 소통하는 웹 브라우저가 `HTTP 클라이언트`이다.
- **웹 서버**는 **웹 리소스를 관리&제공한다.**  (모든 종류의 파일)
  - 다양한 타입을 다룬다. -> 분류가 필요하다 -> **MIME** 타입으로 구분한다.
    - Multipurpose Internet Mail Extensions : 다목적 인터넷 메일 확장 (최초에 이메일 시스템을 통해 전달 되기 위해 개발 그래서 이름에 Mail이 들어감)
- 이런 다양한 타입의 리소스가 있기에 **Uniform Resource Identifier**는 말 그대로 **`통합 자원 식별자`로 정보 리소스를 고유 식별&위치 지정한다.**
  - ex) 지민성의 블로그의 포스트 이미지 https://www.jiminseong.com/_next/image?url=%2Fassets%2Fblog%2Fdaangn-1%2Fcover.png&w=3840&q=75
    - next.js 의 이미지 태그가 활용됨을 알 수 있다.
  
### URI 에는 URN과 URL이 있다.
- URL 부터 설명하자면, `Uniform Resource Locator` 는 `통합 자원 위치지정자`로 리소스의 구체적 **위치**를 나타낸다.
  - 대부분의 URL은 세부분으로 이루어진 표준 포맷을 따른다.
    - 스킴 + 서버 주소 + 서버 리소스
    - 스킴은 프로토콜을 서술하고 http, https가 있다. https는 추후 설명된다.
- URN, `Uniform Resource Name`은 `통합 자원 이름`으로 웹 페이지의 위치(Location)에 상관없이 고유하게 식별하는 **이름**이다.
  - 위치에 영향이 가지 않는 유일 무이한 이름이라고 보면된다. ex) 국제 표준 도서번호 ISBN을 사용하는 책 식별 URN : urn:isbn:12345678
- 하지만 대부분의 URI는 URL이다.

## 트랜잭션
- 웹 클라이언트와 웹 서버는 인터넷이라는 배경 안에서 HTTP 트랜잭션을 구성한다. 
  1. Client : HTTP 요청 메세지 with 명령(메서드), URI -> Server
    - HTTP 메소드는 아래 종류가 있다.
      - `GET` : 이 리소스를 보내줘요.
      - `PUT` : 이 데이터를 요 이름의 리소스로 저장해요.
      - `DELETE` : 이 리소스를 삭제해요.
      - `POST` : 나의 이 데이터를 서버 게이트웨이 애플리케이션으로 보내
        - 게이트웨이(통로) 애플리케이션 이란?
          - 클라이언트와 서버 애플리케이션 간의 통신을 중개하고 보안, 로드 밸런싱, 정책 적용 등의 기능을 수행하는 소프트웨어 또는 하드웨어를 의미. 대표적으로는 웹 애플리케이션의 액세스를 관리하는 `애플리케이션 게이트웨이`, 그리고 클라이언트의 요청을 여러 백엔드 서비스로 라우팅하는 `**API 게이트웨이**` 가 있다.
      - `HEAD` : 요 리소스에 대한 응답에서 HTTP 헤더 부분만 보내줘요.   
  2. Server : HTTP 응답 메시지 with 트랜잭션의 결과
    - 모든 응답은 상태 코드와 함께 반환된다.
    - 메세지 구성은, 시작줄, 헤더, 본문 으로 되어있다.

## TCP 커넥션
- 위에서 말했던 메시지들은 TCP, `Transmission Control Protocol` 즉 전송 제어 프로토콜 커넥션을 통해 이동된다.
-  OSI 7 Layer(국제표준 네트워크 모델)에서의 계층 중 애플리케이션 계층(7단계)을 말한다. (**네트워크 통신의 최상단**)
- HTTP는 애플리케이션 계층 프로토콜(7단계)이다. 
- TCP는 4단계(전송)에 속한다.
- IP는 3단계(네트워크)에 속한다.
- TCP는 **보낸 순서대로 언제든, 어떤 크기의 데이터든, 오류 없이 보낼 수 있음**을 제공한다. 
- 계층이 있기에, TCP 커넥션이 맺어지면 메시지가 **순서가 바뀌거나, 없어지거나, 손상되지** 않는다.

### TCP/IP
- 웹 서버와 웹 클라이언트 사이에 **TCP/IP 커넥션**을 맺어야, 메세지를 전송할 수 있게 된다.

# 1부 2장. URL과 리소스
## URL의 문법
- https://www.jiminseong.com/posts/daangn-1
  - **스킴 : https 이 경우 https 프로토콜을 사용한다.**
  - **서버 위치 : www.jiminseong.com은 서버의 위치이다.**
  - **리소스 경로 : /posts/daangn-1은 리소스의 경로이다.**
- 이렇게 간단하게 구분하기 이전에.. 과거에는 불편함이 있었다.
  - ~~posts/daangn-1이라는 파일을 공유하기 위해서, ftp.jiminseong.com에 FTP로 접속해서 로그인하고 pub 디렉토리~.. ~~

## 상세하게는, URL은 9개로 나뉜다.
  - 스킴, 사용자 이름, 비밀번호, 호스트, 경로, 파라미터(이름/값을 쌍으로 가진다.), 질의(?로 구분), 프래그먼트(#으로 구분, 조각의미)
  - 이때 TCP 프로토콜을 사용하는 HTTP는 기본 포트로 80을 사용한다.
  - 파라미터는 서버에 정확한 요청을 하기 위해 필요한 값을 의미한다.
  - ? 우측에 있는 값들을 질의 컴포넌트라고 한다.
  - #은 프래그먼트에 쓰이며 리소스의 특정 부분을 말한다.
